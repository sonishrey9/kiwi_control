# Release Packaging

## Versioning

Current strategy:

- `0.x` for public beta and rapid iteration
- `-beta.N` suffix for pre-release channels
- aligned versions across `sj-core`, `sj-cli`, and `sj-ui`

## Artifact naming

Release artifact naming is generated by `scripts/prepare-release-manifest.mjs`.

Current patterns:

- `shrey-junior-cli-${version}-${os}-${arch}.tar.gz`
- `shrey-junior-runtime-${version}-${os}-${arch}.tar.gz`
- `shrey-junior-ui-web-${version}-${os}-${arch}.tar.gz`
- `shrey-junior-desktop-${version}-${os}-${arch}.${ext}`

## Distribution targets

- GitHub Releases
- Homebrew formula publication for the CLI
- winget manifests for Windows
- platform bundles for `sj-ui` on Windows, macOS, and Linux

## Signing and updater placeholders

Desktop distribution still requires release-time configuration. Placeholder environment variables are documented in the generated release manifest and signing notes:

- `TAURI_SIGNING_PRIVATE_KEY`
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
- `APPLE_SIGNING_IDENTITY`
- `WINDOWS_CODESIGN_CERT_SHA1`
- `WINDOWS_CODESIGN_PASSWORD`

## Update metadata flow

The intended flow is:

1. Tag a version.
2. Build workspace packages.
3. Generate release manifest metadata.
4. Build CLI and runtime artifacts everywhere.
5. Build signed desktop bundles where signing keys are configured.
6. Publish GitHub Release assets.
7. Update Homebrew and winget manifests to point at the published URLs and checksums.

## Important honesty rules

- do not claim desktop bundles are trustworthy until platform signing is configured
- do not claim updater support is active until the updater manifest and signing are live
- do not claim package-manager availability until the packages are actually published
